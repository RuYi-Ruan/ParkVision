 # ParkVision 中 YOLO 模型对接方案

## 1. 文档目的

本文档用于明确 YOLO 模型在 ParkVision 项目中的职责、调用流程、输入输出定义与后端对接方式，作为后续算法模块开发、前后端联调和毕业设计论文撰写的统一依据。

本方案重点解决以下问题：

1. YOLO 在智能停车系统中具体负责什么。
2. YOLO 如何与 Django 后端和 Vue 前端配合。
3. 系统后续如何从“演示接口”平滑过渡到“真实模型推理”。

---

## 2. 项目背景

本课题为“基于深度学习的住宅小区智能停车系统”。系统目标并不是单纯完成目标检测，而是建设一个完整的停车业务闭环，包括：

- 用户与车辆信息管理
- 车位状态监控
- 车辆进出场识别
- 停车记录生成与查询
- 智能识别结果展示

因此，YOLO 在本项目中承担的是“视觉感知能力”，而不是全部业务能力。

---

## 3. YOLO 在系统中的作用

### 3.1 核心定位

YOLO 的主要作用可以概括为：

**负责从图像中检测出车辆、车牌以及相关目标位置，为后端业务判断提供基础视觉信息。**

### 3.2 不属于 YOLO 的职责

以下内容不应由 YOLO 完成，而应由后端业务系统负责：

- 用户登录与权限控制
- 车辆管理增删改查
- 车位信息维护
- 停车费用计算
- 停车记录写入数据库
- 页面展示与交互

因此，本项目整体职责划分为：

- YOLO：负责检测与定位
- OCR：负责识别车牌文字
- Django：负责业务逻辑、数据存储、接口返回
- Vue：负责管理页面与识别结果展示

---

## 4. YOLO 在本项目中的三个主要应用场景

### 4.1 入口识别场景

在住宅小区入口抓拍图像中，系统需要识别进入车辆的基本信息。

YOLO 在该场景中的作用：

- 检测车辆目标
- 检测车牌区域
- 返回目标框坐标

之后由 OCR 识别车牌号，后端完成以下业务处理：

- 判断车辆是否已登记
- 生成入场停车记录
- 保存入口图片路径
- 更新系统首页统计信息

### 4.2 出口识别场景

在出口抓拍图像中，系统需要确认离场车辆并闭合停车记录。

YOLO 在该场景中的作用：

- 检测当前车辆
- 检测车牌区域
- 为 OCR 提供车牌裁剪区域

后端完成以下业务处理：

- 匹配当前“在场”记录
- 写入出场时间
- 计算停车时长
- 计算或展示停车费用
- 更新停车记录状态

### 4.3 停车区监控场景

在住宅小区停车区画面中，系统需要判断各车位是否被占用。

YOLO 在该场景中的作用：

- 检测画面中的车辆框
- 输出每个车辆的坐标

后端或算法服务再结合预设车位区域，判断：

- 哪些车位被占用
- 哪些车位空闲
- 哪些区域需要维护或人工复核

## 5. 推荐的模型能力拆分

结合本项目需求，建议将算法能力拆分为以下三个部分：

### 5.1 车辆检测模型

功能：

- 输入停车场、入口或出口图像
- 输出车辆框坐标

适用场景：

- 入口抓拍
- 出口抓拍
- 停车区监控

### 5.2 车牌检测模型

功能：

- 输入整图或车辆区域图
- 输出车牌框坐标

适用场景：

- 入口识别
- 出口识别

### 5.3 OCR 车牌识别模块

功能：

- 输入车牌裁剪图
- 输出车牌字符串

推荐说明：

YOLO 更擅长“找目标”，OCR 更擅长“读文字”，因此建议采用：

- YOLO 负责检测车牌区域
- OCR 负责识别车牌字符

该组合更符合工程实践，也更适合毕业设计论文表达。

## 6. 车位状态识别推荐方案

车位状态识别有两种常见方案。

### 方案一：YOLO 检测车辆 + 预定义车位区域

流程：

1. 在停车区图像中使用 YOLO 检测所有车辆。
2. 系统预先保存每个车位的区域坐标。
3. 判断车辆框是否落入某个车位区域。
4. 若车辆框与车位区域重叠度超过阈值，则认定该车位被占用。

优点：

- 实现简单
- 逻辑清晰
- 容易解释
- 更适合毕业设计

### 方案二：YOLO 直接检测车位占用状态

流程：

1. 直接训练模型检测“空车位”“占用车位”等类别。

缺点：

- 标注成本更高
- 数据集要求更高
- 训练与解释难度更大

### 推荐结论

对于当前 ParkVision 项目，推荐使用：

**车辆检测 + 车位区域判断**

这是最稳定、最容易落地、也最适合论文答辩说明的方案。

## 7. 当前项目目录与 YOLO 的对应关系

项目中 `ai/` 目录已经按较合理的方式进行了划分：

### 7.1 models/

负责模型能力本身，例如：

- `vehicle_detector.py`
- `plate_detector.py`
- `ocr_recognizer.py`
- `parking_status.py`

职责说明：

- 加载模型权重
- 执行单次推理
- 输出原始检测结果

### 7.2 services/

负责向 Django 提供组合能力，例如：

- `detect_vehicle.py`
- `detect_plate.py`
- `recognize_plate.py`
- `detect_space_status.py`

职责说明：

- 串联多个模型
- 整理检测结果
- 返回适合业务系统使用的数据结构

### 7.3 utils/

负责辅助能力，例如：

- 图片读写
- 裁剪车牌区域
- 绘制检测框
- 坐标几何计算

## 8. Django 与 AI 模块的推荐调用关系

### 8.1 推荐原则

Django 后端不应该直接处理 YOLO 的底层细节，而应通过 `ai/services/` 提供的高层接口完成调用。

即：

- Django 负责保存图片、调用服务、写入业务结果
- `ai/services/` 负责模型推理和结果整理

### 8.2 推荐调用示例

后端识别接口可以这样调用：

```python
result = recognize_plate(image_path)
```

或：

```python
result = detect_space_status(image_path)
```

这样可以保证：

- Django 代码足够清晰
- 模型替换时影响范围小
- 算法层和业务层职责分离

## 9. 识别接口的推荐处理流程

### 9.1 前端发起识别请求

前端在识别测试页上传图片，请求后端接口：

```text
POST /api/detection/
```

### 9.2 Django 后端处理步骤

推荐流程如下：

1. 接收前端上传的图片
2. 将原始图片保存到 `backend/media/raw/`
3. 调用 `ai/services/` 中的识别服务
4. 获得车辆、车牌、OCR、车位状态等结果
5. 将结果转成统一 JSON 返回前端
6. 如有需要，将识别结果写入停车记录或车位状态表

### 9.3 返回结果示例

```json
{
  "plate": "粤B·A2138",
  "vehicle_type": "car",
  "confidence": 0.96,
  "task": "entry_detection",
  "boxes": [
    {
      "label": "vehicle",
      "x1": 120,
      "y1": 88,
      "x2": 420,
      "y2": 310
    },
    {
      "label": "plate",
      "x1": 198,
      "y1": 244,
      "x2": 286,
      "y2": 279
    }
  ]
}
```

## 10. 当前项目中的 detection 模块如何升级为真实 YOLO 接口

目前 `backend/apps/detection/` 中已经有演示接口骨架，后续可以按照以下步骤升级：

### 第一步：保留当前接口路径不变

保持：

- `GET /api/detection/`
- `POST /api/detection/`

这样前端不需要大改。

### 第二步：替换 detection/services.py 中的演示数据

当前返回的 preset/mock 数据，后续替换为：

- 文件保存逻辑
- 调用 `ai/services/recognize_plate.py`
- 调用 `ai/services/detect_space_status.py`

### 第三步：统一结果格式

无论后续模型如何变化，都建议后端最终向前端返回统一字段：

- `plate`
- `confidence`
- `task`
- `vehicle_type`
- `location`
- `boxes`
- `steps`

这样前端识别展示页可以保持稳定。

## 11. 建议的开发顺序

为了降低接入风险，建议按以下顺序推进：

### 11.1 第一阶段：完成文档与接口约定

目标：

- 明确 YOLO 的职责
- 明确输入输出格式
- 明确 Django 和 AI 服务的边界

### 11.2 第二阶段：先接图片推理

目标：

- 暂不处理视频流
- 先支持上传单张图片完成识别
- 让识别测试页先跑通

原因：

- 图片推理更简单
- 便于调试
- 更适合毕业设计初期演示

### 11.3 第三阶段：接车牌检测 + OCR

目标：

- 实现入口 / 出口识别的核心闭环

### 11.4 第四阶段：接停车区状态判断

目标：

- 将车位监控页切换为真实检测结果

### 11.5 第五阶段：视情况扩展视频流

如果时间充足，再考虑：

- 视频帧抽取
- 实时检测
- 连续帧状态跟踪

对于毕业设计而言，这一步不是最优先。

## 12. 论文撰写时可采用的系统分层描述

在论文中，建议将系统结构描述为三层：

### 12.1 感知层

由 YOLO 与 OCR 组成，负责：

- 车辆检测
- 车牌检测
- 车牌识别
- 车位占用判断

### 12.2 业务层

由 Django + MySQL 组成，负责：

- 用户管理
- 车辆管理
- 车位管理
- 停车记录管理
- 识别结果入库与统计分析

### 12.3 表现层

由 Vue 前端组成，负责：

- 登录与权限界面
- 首页统计展示
- 车位与车辆管理界面
- 停车记录查询
- 识别结果可视化展示
